# nanoGPT 训练数据格式说明

## 为什么使用 bin 格式存储训练数据

在 nanoGPT 中，训练数据被转换为二进制格式（`.bin`）存储，这是深度学习训练中的一个常见优化策略。以下是具体原因：

---

### 1. **存储效率高**

从 `prepare.py` 可以看到，数据被编码为 `uint16` 类型：

```python
train_ids = np.array(train_ids, dtype=np.uint16)
val_ids = np.array(val_ids, dtype=np.uint16)
train_ids.tofile(os.path.join(os.path.dirname(__file__), 'train.bin'))
```

**优势**：
- `uint16` 每个元素仅占 2 字节
- 相比文本格式（每个字符至少 1 字节，加上分隔符等开销），通常节省 50%+ 存储空间
- OpenWebText 的 `train.bin` 约 17GB，如果用文本格式会大得多

---

### 2. **读取速度极快**

二进制文件无需文本解析，可以直接加载为数组：

```python
data = np.memmap(os.path.join(data_dir, 'train.bin'), dtype=np.uint16, mode='r')
```

**对比**：
| 格式 | 读取方式 | 速度 |
|------|----------|------|
| 文本 | 需要逐行解析、分词、转换 | 慢 |
| bin | 直接内存映射，无需解析 | 快 |

---

### 3. **支持内存映射（Memory Mapping）**

这是最关键的优势。`np.memmap` 允许在不将整个文件加载到内存的情况下访问数据：

```python
data = np.memmap(os.path.join(data_dir, 'train.bin'), dtype=np.uint16, mode='r')
```

**优势**：
- 支持处理远超可用内存的数据集（如 17GB 的 train.bin）
- 按需读取，节省内存开销
- 避免一次性加载大文件导致的内存不足问题

---

### 4. **数据已预处理为数值形式**

文本数据需要经过编码才能输入模型：

```python
stoi = { ch:i for i,ch in enumerate(chars) }  # 字符到整数的映射
def encode(s):
    return [stoi[c] for c in s]  # 编码器

train_ids = encode(train_data)  # 文本 → 整数列表
```

**预处理的好处**：
- 训练时无需重复编码，节省计算时间
- 数据已经是模型可直接使用的数值形式
- 编码规则（`stoi`/`itos`）保存在 `meta.pkl` 中，便于后续解码

---

### 5. **便于高效随机访问**

训练时需要随机采样批次数据：

```python
ix = torch.randint(len(data) - block_size, (batch_size,))
x = torch.stack([torch.from_numpy((data[i:i+block_size]).astype(np.int64)) for i in ix])
```

**bin 格式支持**：
- O(1) 时间复杂度的随机访问
- 直接切片操作 `data[i:i+block_size]`
- 相比文本格式需要逐行扫描，效率提升显著

---

### 数据处理流程总结

```
原始文本 (input.txt)
        ↓
   字符级编码 (stoi)
        ↓
   整数列表 (train_ids)
        ↓
   numpy 数组 (uint16)
        ↓
   二进制文件 (train.bin)
        ↓
   内存映射读取
        ↓
   训练批次采样
```

---

### 配套文件说明

| 文件 | 作用 |
|------|------|
| `train.bin` | 训练数据，uint16 编码的 token ID |
| `val.bin` | 验证数据，格式同上 |
| `meta.pkl` | 元信息，包含 `vocab_size`、`stoi`、`itos` |

---

### 总结

使用 bin 格式是权衡存储空间、读取速度和内存使用的最优选择，特别适合大规模语言模型训练场景。这种设计使得 nanoGPT 能够高效处理数十亿 token 的数据集。

## 损失计算机制（estimate_loss 函数）

`estimate_loss` 函数用于估算模型在训练集和验证集上的平均损失，提供对模型性能的准确评估。

### 函数实现

```python
@torch.no_grad()  # 禁用梯度计算，节省内存和计算
def estimate_loss():
    out = {}
    model.eval()  # 切换到评估模式
    for split in ['train', 'val']:
        losses = torch.zeros(eval_iters)  # 存储每次迭代的损失
        for k in range(eval_iters):
            X, Y = get_batch(split)  # 获取一批数据
            with ctx:  # 混合精度上下文
                logits, loss = model(X, Y)  # 前向传播计算损失
            losses[k] = loss.item()  # 记录损失值
        out[split] = losses.mean()  # 计算平均损失
    model.train()  # 切换回训练模式
    return out
```

### 损失计算流程

#### 1. 数据准备
```python
X, Y = get_batch(split)
```
- `X` 是输入序列（如 "Hello, wo"）
- `Y` 是目标序列，比 `X` 右移一位（如 "ello, wor"）

#### 2. 模型前向传播

模型内部的损失计算逻辑：
```python
# 简化版损失计算
logits = self.transformer(x)  # 得到所有位置的预测
loss = F.cross_entropy(logits.view(-1, logits.size(-1)), y.view(-1))
```

**核心原理**：
- 模型输出 `logits` 形状为 `[batch_size, block_size, vocab_size]`
- 将 `logits` 展平为 `[batch_size * block_size, vocab_size]`
- 将目标 `Y` 展平为 `[batch_size * block_size]`
- 使用交叉熵损失计算每个位置预测的平均误差

#### 3. 多次采样求平均

```python
for k in range(eval_iters):  # 默认 eval_iters = 200
    ...
    losses[k] = loss.item()
out[split] = losses.mean()
```

**为什么要多次采样**：
- 单次 batch 的损失存在随机性
- 多次采样取平均可以获得更稳定、更准确的损失估计
- 默认评估 200 个批次，然后计算平均值

### 关键技术点

| 技术 | 作用 |
|------|------|
| `@torch.no_grad()` | 禁用梯度追踪，大幅减少内存占用 |
| `model.eval()` | 关闭 dropout、batch norm 等训练特定行为 |
| `ctx` (混合精度) | 使用 bfloat16/float16 加速计算 |
| `loss.item()` | 将 PyTorch 张量转换为 Python 标量 |
| 多次采样平均 | 降低随机性，提高评估稳定性 |

### 调用时机

在训练循环中，该函数定期被调用：

```python
if iter_num % eval_interval == 0:  # 默认每 2000 步评估一次
    losses = estimate_loss()
    print(f"step {iter_num}: train loss {losses['train']:.4f}, val loss {losses['val']:.4f}")
```

### 评估结果解读

| 情况 | 含义 |
|------|------|
| `train loss` 低，`val loss` 低 | 模型拟合良好 |
| `train loss` 低，`val loss` 高 | 模型过拟合 |
| `train loss` 高，`val loss` 高 | 模型欠拟合 |

该函数是训练过程中监控模型性能的核心工具。

## 数据集数据量比较

### 各数据集规模对比

| 数据集 | 训练集 token 数 | 验证集 token 数 | 总 token 数 |
|--------|----------------|----------------|-------------|
| **openwebtext** | ~9B (9,035,582,198) | ~4M (4,434,897) | ~9.04B |
| **shakespeare_char** | 1,003,854 | 111,540 | 1,115,394 |
| **shakespeare** | 301,966 | 36,059 | 338,025 |

### 结论

**数据量最少的数据集是 `shakespeare`**（tiny shakespeare）。

该数据集包含：
- 训练集：301,966 个 token
- 验证集：36,059 个 token
- 总计：约 33.8 万 token

### 适用场景

| 数据集 | 特点 | 适用场景 |
|--------|------|----------|
| **openwebtext** | 大规模（~9B token） | 预训练大型语言模型 |
| **shakespeare_char** | 中等规模，字符级 | 字符级语言模型实验、快速原型开发 |
| **shakespeare** | 最小规模 | 快速测试、教学演示、算法验证 |

`shakespeare` 数据集因其小巧的规模，非常适合用于快速验证模型架构和训练流程是否正常工作。
